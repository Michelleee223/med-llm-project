## 一、项目整体架构

我的医疗问答大模型项目包含以下几个核心模块：

**4-med-prepare-dataset** 负责数据集的下载、清洗和格式转换。

**5-med-eval** 负责模型评测，用来对比原版模型和微调版模型的性能差异。

**6-med-fine-tuning-llm** 负责文本大语言模型的微调，基于纯文本的医疗问答数据进行训练。

**7-med-fine-tuning-vl** 负责视觉语言模型的微调，基于医学报告图片和文字说明数据进行训练。

**8-med-service** 负责模型服务的部署，包括 vLLM 推理引擎的启动脚本和 FastAPI 网关，以及一个 Web 前端页面。

除此之外，项目中还有三个模型权重目录。**qwen25-14b** 是 Qwen2.5-14B 原始模型权重目录，**qwen25-14b-finetuned-lora** 是该模型经过 LoRA 微调后的适配器权重目录，**qwen25-vl-7b** 是 Qwen2.5-VL-7B 原始模型权重目录，**qwen25-vl-7b-finetuned** 是该模型经过全量微调并合并权重后的完整模型目录。

整个项目走了两条技术路线。第一条是 LLM 路线，基于 Qwen2.5-14B，使用 LoRA 微调技术，专注于医疗领域的问答对话生成。第二条是 VL 路线，基于 Qwen2.5-VL-7B，使用全量微调，专注于医学报告图片的解读和异常指标识别。

## 二、数据集准备

### 2.1 数据来源

原始数据来自 cMedQA2 数据集，这是一个中文医疗问答数据集，从 Gitee 平台下载得到。下载后得到两个压缩包，分别是 question.zip 和 answer.zip，解压后得到两个 CSV 文件。

这两个 CSV 文件通过一个叫 question_id 的字段关联在一起。question.csv 文件中每行记录包含两个字段，一个是 question_id，代表问题的唯一编号，另一个是 content，代表用户的提问内容。answer.csv 文件中每行记录同样包含 question_id 和 content 两个字段，其中 content 代表医生的回答内容。

### 2.2 数据集清洗与划分

我编写了一个 Python 脚本来处理原始数据。这个脚本的工作分为四个步骤。

第一步是读取两个 CSV 文件。使用 Python 的 csv 模块中的 DictReader 方法，以字典的方式读取每一行数据。读取完成后，将 question.csv 中的所有数据存入一个以 question_id 为键、content 为值的字典中，同样的操作对 answer.csv 也做一遍，得到另一个字典。

第二步是合并问答对。取两个字典的键的交集，也就是说找到那些既在 question.csv 中出现又在 answer.csv 中出现的 question_id。然后根据这些共同的键，将问题和答案配对合并，形成一个包含问答内容的字典。

第三步是转换为标准对话格式。我把每一对问答转换为一个标准的对话格式，每条记录包含一个 conversations 列表，列表中有两条消息。第一条消息的 role 是 user，内容是用户的提问。第二条消息的 role 是 assistant，内容是医生的回答。这样的格式是训练大语言模型时最常用的标准格式。

第四步是按比例划分数据集。我把所有合并好的问答对按顺序等分为三份。第一份作为训练集，第二份作为验证集，第三份作为测试集。这样就得到了三个 JSON 文件，分别是 med-dataset-train.json、med-dataset-valid.json 和 med-dataset-test.json。

输出的 JSON 文件中，每条记录的结构是这样的：最外层是一个对象，这个对象有一个 key 叫 conversations，它的值是一个数组，数组中包含两个对象，第一个对象的 role 是 user，content 是用户的问题，第二个对象的 role 是 assistant，content 是医生的回答。

### 2.3 转换为评测格式

为了后续能够用评测框架来评估模型性能，我还需要把数据转换为另一种格式。这是因为评测框架需要的是特定的输入输出格式，而不是我训练时用的对话格式。

我编写了一个转换脚本，它的功能是读取刚才生成的 JSON 文件，遍历其中的每一条对话记录，从中提取出用户的问题作为 query 字段，提取出助手的回答作为 response 字段，然后将每一对这样的数据写成 JSONL 格式的一行。JSONL 格式就是每行一个 JSON 对象，这样便于评测框架逐行读取和处理。

转换脚本会跳过那些格式不正确的对话项，比如缺少 user 消息或者 assistant 消息的记录，确保输出的数据都是完整有效的。

### 2.4 执行流程

数据准备阶段需要按顺序执行两个脚本。首先解压下载好的数据压缩包，然后运行处理脚本来完成数据的读取、合并和划分。之后再运行转换脚本，把验证集数据转换为评测框架需要的 JSONL 格式。完成这些步骤后，数据集的准备工作就结束了，可以进入模型评测和微调阶段。

## 三、模型评测

### 3.1 评测工具与指标

我使用 EvalScope 作为评测工具。EvalScope 是modelscope平台提供的一个评测框架，支持以 SERVICE 模式对模型进行评测。在这种模式下，模型通过 OpenAI 兼容的 API 接口提供服务，评测框架通过 HTTP 请求调用这个接口来获取模型的回答，然后与标准答案进行对比打分。

评测用的数据是 cMedQA2 数据集的验证集，一共 256 条数据。评测时使用的指标有两类，分别是 BLEU 和 ROUGE。BLEU 指标衡量生成文本与参考答案在词级 N-gram 上的精确率重叠程度，数字越大表示重叠越多。ROUGE 指标则衡量生成文本与参考答案在词序列层面的召回率，R 代表 Recall（召回率），P 代表 Precision（精确率），F 代表 F1（调和平均值）。

我分别对原版 Qwen2.5-14B 和微调后的 Qwen2.5-14B 进行了评测，这样可以直接对比微调带来的效果提升。

### 3.2 原版模型评测

首先需要对原版模型进行评测，作为基线参考。

评测的第一步是启动 vLLM 推理服务。vLLM 是一个高效的大语言模型推理引擎，支持 OpenAI 兼容的 API。我用 vLLM 来托管原版的 Qwen2.5-14B 模型，这样评测脚本就可以通过 HTTP 请求来调用它。

启动服务时需要指定很多参数，我来逐一说明。host 设置为 0.0.0.0，表示服务监听所有网络接口。port 设置为 8000，是访问这个服务的端口号。trust-remote-code 设置为真，允许加载模型作者自定义的代码。max-model-len 设置为 4096，表示模型最大支持 4096 个 token 的上下文长度。max-num-batched-tokens 设置为 32768，表示每次最多处理 32768 个 token。max-num-seqs 设置为 32，表示每次最多同时处理 32 个序列。dtype 设置为 auto，让框架自动选择合适的数据类型。kv-cache-dtype 设置为 auto，让框架自动选择 KV 缓存的数据类型。tensor-parallel-size 设置为 1，表示只使用一块 GPU。gpu-memory-utilization 设置为 0.90，表示 GPU 显存占用 90%，留出 10% 给其他开销。

启动服务后，vLLM 会自动加载模型权重到显存中，然后等待 HTTP 请求。当收到请求时，它会使用 vLLM 特有的 PagedAttention 技术来高效地处理推理任务。

第二步是编写评测配置文件。我定义了一个 TaskConfig 对象，其中 model 指定为模型路径，api_url 指定为刚才启动的 vLLM 服务的地址，eval_type 指定为 SERVICE 模式，datasets 指定为 general_qa，这是评测框架内置的一种问答评测格式，limit 指定为 256，意味着评测 256 条数据。

评测框架会依次执行以下步骤：加载评测数据集，向 vLLM 服务发送请求获取模型回答，计算 BLEU 和 ROUGE 分数，最后汇总生成评测报告。

### 3.3 微调后模型评测

评测完原版模型后，我对微调后的模型也进行了同样的评测。

微调后模型的服务启动命令与原版基本相同，但有两个关键区别。第一个是多了 --enable-lora 参数，表示启用 LoRA 适配器功能。第二个是多了 --lora-modules 参数，指定了 LoRA 适配器的模块名和路径，比如 mylora 等于微调产出的 LoRA 适配器目录。这样 vLLM 在加载模型时，会同时加载基础模型和 LoRA 适配器，在推理时将两者的计算结果合并。

评测配置文件中，model 字段仍然指向基础模型路径，但实际推理时会自动应用 LoRA 适配器。

### 3.4 评测结果对比

原版 Qwen2.5-14B 的评测结果如下。BLEU-1 是 0.108，BLEU-2 是 0.0131，BLEU-3 和 BLEU-4 接近于零。ROUGE-1 的 Recall 是 0.34，Precision 是 0.1067，F1 是 0.1552。ROUGE-2 的 F1 是 0.0214。ROUGE-L 的 Recall 是 0.2905，F1 是 0.1095。

微调后 Qwen2.5-14B 的评测结果如下。BLEU-1 是 0.1635，相比原版提升了 51.4%。BLEU-2 是 0.0358，相比原版提升了 173.3%。ROUGE-1 的 Recall 是 0.2135，相比原版有所下降，但 Precision 提升到了 0.3054，F1 提升到了 0.2337，相比原版提升了 50.6%。ROUGE-L 的 F1 提升到了 0.1857，相比原版提升了 69.6%。

从这些数据中我可以得出几个关键发现。BLEU-1 和 BLEU-2 的提升非常显著，说明微调后模型生成文本中与参考答案的词级重叠度大幅提高。ROUGE-1-F 和 ROUGE-L-F 也有明显提升，说明生成质量更接近参考答案。值得注意的是，ROUGE-Recall 略有下降，这是因为微调后模型倾向于生成更简洁、风格更统一的回答，覆盖率略低但精确率更高。整体结论是，微调后模型在医疗问答上的生成质量显著优于原版。

评测耗时方面，原版模型评测大约花了 21 分钟，微调版模型评测大约花了 8 分钟。这是因为评测时 vLLM 已经缓存了权重，不需要重复加载。

## 四、LLM 微调

### 4.1 技术框架选择

我选择使用 Unsloth 加速库来进行 LoRA 微调。Unsloth 是一个专门为微调设计的高效框架，它通过一系列优化手段，可以实现比标准训练快两倍的速度，同时节省一半的显存。这是一个非常重要的技术选型决策，因为它直接决定了微调过程的效率和成本。

Unsloth 的核心优化包括：对矩阵运算进行手写优化、使用更高效的注意力机制实现、以及智能的显存管理。在我的项目中，我大量使用了 FastLanguageModel、SFTTrainer、TrainingArguments 等组件，它们都来自 Unsloth 及其依赖的 TRL 库。

### 4.2 模型加载

加载预训练模型是微调的第一步。我使用 FastLanguageModel.from_pretrained 方法来加载 Qwen2.5-14B 模型。

加载时需要指定多个参数。model_name 指向本地模型权重目录，由于模型已经提前下载好，我设置了 local_files_only 为真，确保不会尝试联网下载。dtype 设置为 None，让框架自动选择最合适的数据类型，通常会是 bf16。max_seq_length 设置为 2048，表示最大序列长度是 2048 个 token。load_in_4bit 和 load_in_8bit 都设为假，表示不使用量化，以全精度进行训练。full_finetuning 设为假，表示不使用全量微调，而是使用 LoRA。use_gradient_checkpointing 设为真，表示开启梯度检查点技术，通过在反向传播时重新计算中间结果来节省显存。

### 4.3 LoRA 配置详解

LoRA 是 Low-Rank Adaptation 的缩写，中文翻译是低秩适配。这是一种非常高效的模型微调技术，其核心思想是在预训练模型的权重旁边添加两个小的矩阵 A 和 B。原始权重矩阵记作 W，在前向传播时，实际使用的是 W 加上 BA 的结果，其中 A 和 B 是新添加的可训练矩阵。由于 A 和 B 的秩远小于 W，所以参数量也远小于 W，从而实现了只训练少量参数就达到微调效果的目的。

在我的配置中，LoRA 的秩 r 设置为 16。16 是一个经验值，在效果和效率之间取得了良好平衡。数值越高表达能力越强，但参数量和计算量也越大。

我设置了 target_modules 参数，指定了所有需要应用 LoRA 的模块名称。这些模块包括 q_proj、k_proj、v_proj、o_proj，也就是注意力机制中的查询、键、值和输出投影矩阵，以及 gate_proj、up_proj、down_proj，也就是 FFN（前馈神经网络）层中的门控和上投影、下投影矩阵。将 LoRA 应用到这些模块，可以全面调整模型的行为。

lora_alpha 设置为 16，通常设置为 rank 的两倍，这个值影响 LoRA 权重对原有权重的缩放程度。lora_dropout 设置为 0，表示不使用 dropout。bias 设置为 none，表示不对偏置项进行训练。use_gradient_checkpointing 设置为 unsloth，让 Unsloth 框架自动应用梯度检查点优化。

### 4.4 数据集处理

数据集加载完成后，需要对它进行格式化处理，使其符合 Qwen 模型的输入要求。

我首先定义了一个 chat template，也就是聊天模板。这个模板包含了一个系统提示词，告诉模型它是一个专业的医疗健康顾问，能够提供有关体检指标异常的专业健康建议。然后用 ### 指令: 和 ### 回应: 的格式来包裹用户输入和模型输出。

然后使用 apply_chat_template 方法，将这个模板应用到数据集中的每一条记录上。这个方法会自动在用户消息前面加上系统提示词，在回复前面加上格式标记，并使用 Qwen 模型特有的特殊 token 来标记不同角色的消息。

### 4.5 训练超参数

训练时的各项超参数设置如下。per_device_train_batch_size 设置为 64，表示每块 GPU 每一步处理的样本数是 64。gradient_accumulation_steps 设置为 4，表示每 4 步才进行一次梯度更新，所以实际的有效批大小是 64 乘以 4 等于 256。warmup_steps 设置为 5，表示前 5 步是预热阶段，学习率从零逐渐上升到设定的值。num_train_epochs 设置为 1，表示整个训练集过一遍。learning_rate 设置为 2e-4，即 0.0002，这是 LoRA 微调中常用的学习率。bf16 设置为真，表示使用 bf16 混合精度训练，bf16 比 fp16 更稳定，是目前大模型训练的标准精度。optim 设置为 adamw_8bit，表示使用 8 位 AdamW 优化器，可以节省 75% 的优化器显存占用。weight_decay 设置为 0.01，用于防止过拟合。lr_scheduler_type 设置为 linear，表示学习率线性衰减。seed 设置为 3407，确保实验可复现。output_dir 设置为 outputs，训练日志和检查点会保存到这个目录。report_to 设置为 none，表示不将训练日志上报到任何平台。

### 4.6 推理验证

训练完成后，我立即用一条测试输入来验证模型效果。

我使用 FastLanguageModel.for_inference 方法将模型切换到推理模式，然后构造一条测试消息，内容是"我肚子疼"。将这条消息通过聊天模板转换为模型需要的输入格式，然后使用 TextStreamer 进行流式推理。流式推理的好处是，模型生成的内容会实时打印出来，不需要等待全部生成完毕。

生成时设置了 max_new_tokens 为 1024，表示最多生成 1024 个新 token。pad_token_id 设置为 tokenizer 的 eos_token_id，用来标记序列结束。

### 4.7 保存模型

训练完成后，使用 save_pretrained 方法将 LoRA 适配器保存到指定目录。这个方法只会保存 LoRA 相关的权重，也就是新增的 A 和 B 矩阵，不会保存完整的模型权重，因此生成的文件非常小。只有几 MB 到几十 MB，而完整模型通常有几十 GB。

### 4.8 其他模型微调尝试

除了 Qwen2.5-14B，我还尝试了对其他模型进行微调。

第一个是 MedGemma 1.5 4B 微调。这是一个专门针对医疗领域的小型模型，只有 40 亿参数。我使用 gemma-3 的模板格式来适配这个模型。在 LoRA 配置上，我只开启了语言层的微调和 MLP 层的微调，视觉层和注意力模块保持冻结，因为这个模型虽然有视觉能力，但我的任务主要是文本问答，不需要微调视觉部分。训练步数设置为 240 步，每块 GPU 的批大小为 2。另外我还使用了 train_on_responses_only 这个技术，它的作用是只让模型在回复部分计算损失，而不让模型在用户指令部分学习，这样可以让模型更好地学习如何回答问题，而不是学习如何重复指令。

第二个是 OSS 20B 微调。这是一个 200 亿参数的大模型。我使用 standardize_sharegpt 方法来统一数据集格式，因为这个模型要求特定的数据格式。同样使用了 train_on_responses_only 技术，并指定了这个模型特有的分隔符格式。训练步数设置为 90 步，每块 GPU 的批大小为 1。

### 4.9 执行流程

微调的执行流程很简单。设置一系列环境变量来确保离线运行和禁用不必要的功能，然后运行训练脚本即可。这些环境变量包括 HF_HUB_OFFLINE 设置为 1，强制离线模式不联网下载；HF_HUB_DISABLE_TELEMETRY 设置为 1，禁用遥测数据上报；HF_DEBUG 设置为 1，开启调试模式方便排查问题。

## 五、VL 微调

### 5.1 背景与目标

Qwen2.5-VL-7B 是一个多模态视觉语言模型，它既能够理解文本，也能够理解图片。相比 Qwen2.5-14B，它多了一个视觉编码器，可以处理图像输入。

我的目标是让这个模型专门学会解读医学检测报告的图片。当用户上传一张医学检测报告的图片时，模型应该能够识别出其中的异常指标，并给出简洁的说明。

### 5.2 技术方案选择

这里有一个非常重要的技术决策点。vLLM 推理引擎对 VL 模型不支持单独加载 LoRA 适配器。这意味着，如果我像 LLM 那样只保存 LoRA 适配器，vLLM 无法单独加载它。

为了解决这个问题，我选择了全量微调加权重合并的方案。具体做法是，在原模型基础上进行 LoRA 微调，训练完成后用 save_pretrained_merged 方法将 LoRA 权重合并回原模型，生成一个包含完整权重的模型目录。部署时直接使用这个合并后的完整模型，不需要任何 LoRA 相关的加载选项。

这个方案的缺点是合并后的模型权重和原模型一样大，无法享受 LoRA 的轻量优势。但对于 VL 模型来说，这是目前唯一可行的方案。

### 5.3 模型加载

加载 VL 模型时，使用 FastVisionModel 而不是 FastLanguageModel。FastVisionModel 是专门处理视觉语言任务的版本，它会自动处理图像相关的配置。

加载参数与 LLM 类似，但有一个区别是 use_gradient_checkpointing 设置为假。这是因为 VL 模型的显存需求非常大，视觉编码器本身就要占用大量显存，加上语言解码器的 LoRA，显存已经接近饱和，无法再开启梯度检查点来换取更多的显存空间。

### 5.4 VL 模型的 LoRA 配置

VL 模型的 LoRA 配置比 LLM 更加全面。我设置了 finetune_vision_layers 为真，开启视觉编码器的微调。这是必要的，因为要理解医学报告中的图像内容，必须让模型学会识别各种医学指标和异常情况。finetune_language_layers 也设为真，微调语言解码器，确保模型能够用准确的中文来描述图像内容。finetune_attention_modules 设为真，微调注意力模块。finetune_mlp_modules 设为真，微调 MLP 层。

LoRA 的秩 r 和 alpha 仍然是 16 和 16，与 LLM 配置保持一致。

### 5.5 数据集构建

VL 模型的数据集与 LLM 完全不同。LLM 的数据是纯文本的问答对，而 VL 的数据是图片加文字的配对。

我使用 imagefolder 格式来组织数据集，也就是将所有训练图片放在一个文件夹中，框架会自动读取这些图片文件。

每张图片对应一条训练数据，有固定的指令内容："解读这个医学报告，一句话指出其中异常指标"。训练时，系统会构造一个对话，对话中用户的输入由两部分组成，一部分是这条指令文本，另一部分是图片本身。助手的回复则是图片对应的标准答案，也就是这张报告中的异常指标说明。

图像的预处理工作包括等比缩放。如果图片的最长边超过 1280 像素，就按照 1280 比最长边的比例进行缩放，保持宽高比不变。缩放后的图片保存为 JPEG 格式。之所以选择 1280 这个数值，是因为它能在图像清晰度和 token 数量之间取得良好平衡。

### 5.6 VL 训练配置

VL 模型的训练配置与 LLM 有几个显著区别。

必须使用 UnslothVisionDataCollator 作为数据整理器。这是专门为视觉语言任务设计的数据整理器，负责处理图像的批处理和编码。

remove_unused_columns 设置为假，表示保留所有列，包括图像列。如果设为真，数据框架可能会在预处理时丢弃图像列，导致训练失败。

dataset_kwargs 中设置了 skip_prepare_dataset 为真，跳过数据集的自动预处理步骤，由我手动控制数据的格式。

num_train_epochs 设置为 4，比 LLM 训练的 1 个 epoch 多很多。这是因为 VL 模型需要更多的训练轮次才能学会理解图像中的医学指标。

max_length 设置为 2048，比 LLM 的 2048 相同，但 VL 模型的 token 消耗更大，因为图像编码会占用大量 token。

per_device_train_batch_size 设置为 2，比 LLM 的 64 小很多。这是因为 VL 模型处理图像需要大量显存，每块 GPU 只能放下很小的批大小。gradient_accumulation_steps 仍然为 4，所以有效批大小是 8。

### 5.7 显存监控

VL 训练对显存的要求非常高。在训练开始前，我会记录当前的 GPU 显存使用情况，然后在训练过程中和训练结束后再次记录，用来计算训练实际消耗了多少显存。输出信息包括 GPU 型号、最大显存、总使用显存、LoRA 部分消耗的显存、以及占总显存的百分比。这些信息对于判断训练是否能够成功运行非常重要。

### 5.8 权重合并

训练完成后，最关键的一步是权重合并。使用 save_pretrained_merged 方法，将 LoRA 适配器的权重合并到原模型中，输出为一个完整的、可部署的模型目录。这个合并后的模型不依赖任何 LoRA 加载选项，可以直接用标准的 vLLM 来加载和推理。

### 5.9 验证流程

微调完成后，我需要验证模型是否学会了正确的行为。

验证流程分为两步。第一步是启动 vLLM 推理服务，使用合并后的完整模型。需要注意设置 max-model-len 为 8k，因为 VL 模型需要更长的上下文来容纳图像编码。同时需要将 OMP_NUM_THREADS 环境变量设为 1，避免多线程冲突。gpu_memory_utilization 设置为 0.90，给 VL 模型分配更多显存。

第二步是运行验证脚本。这个脚本会读取一张测试用的医学报告图片，对图片进行缩放处理，将缩放后的图片转换为 base64 编码，然后通过 HTTP 请求发送到 vLLM 服务的 /v1/chat/completions 端点。请求中指定模型名称和对话内容，然后解析返回的 JSON 响应，提取出模型生成的异常指标描述。

图片缩放的具体做法是：用图像处理库打开图片，检测图片的宽高，如果最长边超过 1280，就按比例缩小到 1280，然后将缩放后的图片以 JPEG 格式编码为 base64 字符串。之所以将图片转为 base64，是因为模型 API 通过 HTTP 传输图像内容，base64 是最通用的编码方式。

请求格式遵循 OpenAI 的 vision 规范。用户消息的 content 是一个数组，数组中包含两个元素，第一个是文本类型的元素，内容是分析指令"解读这个医学报告，一句话指出其中异常指标"，第二个是 image_url 类型的元素，内容是图片的 base64 数据，需要包装在 data:image/jpg;base64, 的前缀后面。

## 六、服务部署

### 6.1 服务架构设计

整个服务系统采用三层架构设计。

最前端是一个 Web 界面，用户在浏览器中打开页面，可以上传医学检测报告的图片。Web 界面运行在 8080 端口，基于 FastAPI 框架构建。

中间层是 FastAPI 网关，负责接收用户请求、预处理图片、调度下游模型、汇总结果。当用户上传图片时，FastAPI 先对图片进行缩放，然后同时向两个 vLLM 服务发送请求，等待两个结果返回后，将它们组装成最终的 JSON 响应返回给前端。

最后是 vLLM 推理层，包含两个独立的推理服务。第一个是 LLM 推理服务，运行在 8001 端口，托管着 Qwen2.5-14B 加 LoRA 适配器的组合，负责根据 VL 模型的输出生成健康建议。第二个是 VL 推理服务，运行在 9002 端口，托管着 Qwen2.5-VL-7B 全量微调后的完整模型，负责分析医学报告图片并识别异常指标。

这种双模型流水线设计的核心理念是分工合作。VL 模型负责"看"，LLM 模型负责"想"。VL 模型从图像中提取出异常指标信息，这是一段结构化的文字描述。然后 LLM 模型基于这段描述，生成具体的、面向用户的健康建议。分开的好处是，每个模型都可以专注于自己的任务，不需要同时兼顾图像理解和文字生成两个能力，从而达到更好的效果。

### 6.2 vLLM 服务启动

LLM 推理服务的启动命令中，关键参数包括：端口为 8001，enable-lora 为真表示启用 LoRA 功能，lora-modules 指定了 LoRA 适配器的模块名，模块名为 mylora，这个名称在后续 API 调用时需要使用。gpu_memory_utilization 设置为 0.6，比评测时的 0.9 更保守，因为服务运行时需要预留一些显存给请求处理。

VL 推理服务的启动命令中，关键参数包括：端口为 9002，模型路径指向合并后的完整 VL 模型，不需要 LoRA 相关的参数。max-model-len 设置为 8k，即 8192，这是为了容纳图像编码产生的大量 token。gpu_memory_utilization 设置为 0.3，比 LLM 服务更保守，因为 VL 模型本身更大，需要更多显存空间。

两个服务都使用了多个性能调优参数。max-num-batched-tokens 设置为 32768，max-num-seqs 设置为 32，这些参数控制了 vLLM 的批处理能力和并发处理能力。dtype 和 kv-cache-dtype 都设为 auto，让框架自动选择最优的数据类型配置。trust-remote-code 设为真，允许加载模型自定义代码。

### 6.3 FastAPI 服务演进

FastAPI 服务经历了四个版本的迭代，每个版本都是在上一版基础上增加新功能。

第一版是最基础的版本，只实现了文件上传功能。接收用户上传的图片文件，然后返回一些基本信息，比如文件名、文件大小。这个版本的作用是验证文件上传管道是否正常工作，以及静态文件服务是否配置正确。

第二版在第一版基础上增加了图片处理功能。接收图片文件后，自动将图片缩放到最长边不超过 1280 像素，然后返回 base64 编码后的图片数据。这个版本的作用是验证图片预处理流程是否正确，以及 base64 编码是否工作正常。

第三版集成了 vLLM 调用，这是最关键的版本。接收图片文件后，首先进行缩放和 base64 编码，然后依次调用 VL 模型和 LLM 模型，最后将两个模型的结果汇总返回。具体流程是，先向 9002 端口的 VL 服务发送请求，传入 base64 图片和分析指令，获取异常指标描述，然后再向 8001 端口的 LLM 服务发送请求，传入 VL 的输出，获取健康建议。两个请求使用异步 HTTP 客户端发送，timeout 设置为 60 秒。

第四版在第三版基础上添加了 Web 前端界面。通过 FastAPI 的 StaticFiles 功能，将前端文件目录挂载到根路径下，并启用 html 模式，让 FastAPI 自动处理 SPA 应用的路由。

### 6.4 核心 API 逻辑

FastAPI 的核心 API 是一个 POST 端点，路径是 /image。

接收请求时，首先检查是否真的上传了图片。如果上传数据为空或过小，直接返回错误。然后对图片进行缩放处理，将最长边调整为 1280 像素，保持宽高比。缩放后再进行一次 base64 编码。

接下来进入双模型调用阶段。首先构建 VL 模型的请求 payload。这个请求指定模型为 qwen25-vl-7b-finetuned，也就是合并后的 VL 模型。用户消息中包含两条内容：一条文本指令，要求模型仔细分析医学检测报告图片，识别异常指标，如果没有异常则明确说明；另一条是 base64 编码的图片数据，使用 image_url 格式传输。max_tokens 设置为 256，控制 VL 模型输出的最大长度。

将这个请求发送到 VL 服务的 /v1/chat/completions 端点，解析返回的 JSON，提取出 VL 模型的输出内容，这就是异常指标描述。

然后构建 LLM 模型的请求 payload。这个请求指定模型为 mylora，也就是 LoRA 适配器的名称。用户消息的内容是将 VL 的输出包装在 prompt 中，要求 LLM 基于这些异常指标提供健康建议和注意事项，包括饮食、运动和生活方式等方面的指导。max_tokens 设置为 512，比 VL 模型更长，因为健康建议通常比指标描述更详细。

将这个请求发送到 LLM 服务的 /v1/chat/completions 端点，解析返回的 JSON，提取出 LLM 模型的输出内容，这就是健康建议。

最后将两张图片的大小信息、VL 输出和 LLM 输出组装成 JSON 响应返回给前端。

### 6.5 前端页面设计

Web 前端界面包含三个主要部分：头部区域、聊天区域和输入区域。

头部区域显示"医疗报告助手"的标题，使用医疗主题的渐变蓝色背景。

聊天区域是整个界面的核心，显示欢迎消息和对话记录。每条记录分为两种样式。用户消息显示在右侧，使用蓝色渐变背景的气泡。服务器消息显示在左侧，使用白色背景的气泡，内部又分为两个区块，分别是"检测结果分析"和"健康建议"，分别对应 VL 模型和 LLM 模型的输出。

服务器消息的显示支持 Markdown 格式。前端实现了一个简单的 Markdown 解析器，可以处理一到六级标题、加粗文本、斜体文本、无序列表、有序列表和换行符等常用格式。

加载状态时，显示一个三色脉冲动画和一个秒表计时器，实时显示已经等待了多少秒。这个设计让用户知道系统正在处理中，而不是请求卡死了。

输入区域包含一个图片上传按钮和一个发送按钮。上传按钮平时是虚线边框，上传图片后变成实线边框并显示预览缩略图。发送按钮平时是灰色的不可点击状态，上传图片后变为蓝色可点击状态。点击发送后，图片预览消失，按钮变回不可点击状态。

错误处理方面，网络错误、超时错误和服务器错误都有对应的错误消息显示。超时阈值设置为 600 秒，也就是 10 分钟，对于图片分析和生成来说足够充裕。

### 6.6 启动顺序

部署整个服务需要按正确的顺序启动各个组件。

首先启动 LLM 推理服务，监听 8001 端口。然后启动 VL 推理服务，监听 9002 端口。最后启动 FastAPI 网关，监听 8080 端口。Web 前端直接由 FastAPI 提供，无需单独启动。

前端页面会通过 JavaScript 发送 HTTP 请求到 FastAPI 的 /image 端点，请求通过 form-data 格式携带图片文件。FastAPI 收到后进行上述的双模型调用流程。

## 七、项目总结

### 7.1 技术栈汇总

整个项目使用的技术栈分为几个层次。

在预训练模型层面，使用了 Qwen2.5-14B 作为文本生成的基础模型，使用了 Qwen2.5-VL-7B 作为多模态视觉语言模型。

在微调框架层面，使用了 Unsloth 作为高效的微调加速库，使用了 TRL 库中的 SFTTrainer 作为训练器，使用了 PEFT 库中的 LoRA 技术实现参数高效微调。

在推理引擎层面，使用了 vLLM 作为模型服务的高效推理引擎，它提供了 OpenAI 兼容的 API 接口，支持 PagedAttention 和连续批处理等优化技术。

在服务框架层面，使用了 FastAPI 作为 RESTful API 网关，使用了 uvicorn 作为 ASGI 服务器，使用了 httpx 作为异步 HTTP 客户端。

在前端层面，使用了原生的 HTML、CSS 和 JavaScript，没有引入任何框架依赖，实现了完整的图片上传、预览、发送和结果显示功能。

在评测工具层面，使用了 EvalScope 评测框架，使用了 BLEU 和 ROUGE 作为评测指标。

### 7.2 四项关键决策

第一项决策是 LoRA 与全量微调的选择。LLM 模型使用 LoRA 微调，原因是保留预训练模型的知识基础，同时以极低的参数量和显存开销实现领域适配。VL 模型使用全量微调，因为 vLLM 对 VL 模型的 LoRA 加载有限制，必须合并权重后才能正常推理。两条路线各有利弊，但最终都达到了预期的效果。

第二项决策是评测指标的选择。选择 BLEU 和 ROUGE 作为评测指标，而不是准确率之类的分类指标，原因是医疗问答任务本质上是一个文本生成任务，模型生成的回答不可能和标准答案完全一致，需要用相似度指标来衡量质量。BLEU 衡量精确率，ROUGE 衡量召回率，两者结合可以全面评估生成质量。

第三项决策是 VL 权重合并策略。由于 vLLm 不支持 VL 模型单独加载 LoRA 适配器，我采用了 save_pretrained_merged 方法将 LoRA 权重合并回原模型，生成一个完整的模型目录。这种方式的代价是模型体积恢复到原始大小，但换来了推理时的便利性和稳定性。

第四项决策是双模型流水线架构。将 VL 模型和 LLM 模型分开，让 VL 专注于看图识别异常指标，LLM 专注于基于异常指标生成建议。这种分工设计比用一个 VL 模型同时完成看图和生成两个任务效果更好，因为每个模型都专注于自己擅长的领域。

### 7.3 评测结果总结

微调后 Qwen2.5-14B 在 cMedQA2 医疗问答数据集上取得了显著的进步。BLEU-1 分数从 0.108 提升到了 0.1635，提升幅度为 51.4%。BLEU-2 分数从 0.0131 提升到了 0.0358，提升幅度为 173.3%。ROUGE-1-F 分数从 0.1552 提升到了 0.2337，提升幅度为 50.6%。ROUGE-L-F 分数从 0.1095 提升到了 0.1857，提升幅度为 69.6%。

这些数据充分证明了 LoRA 微调策略在医疗领域问答任务上的有效性。通过在 cMedQA2 医疗问答数据集上的微调，Qwen2.5-14B 学会了医疗领域的专业表达方式和知识体系，能够生成更符合医疗场景的回答。
